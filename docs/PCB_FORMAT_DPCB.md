# PCB FORMAT: .dpcb
# Last updated: 2026-03-09
# Project: AI-Driven KiCad PCB Design Pipeline


## OVERVIEW

The .dpcb format is a compact, text-based representation of a PCB
layout. It contains the complete board state — not diffs. Every
iteration produces a new complete .dpcb file.

This is the AI's working format. It is optimised for AI readability
and manipulation — a full board state in ~50-100 lines versus
thousands of lines in .kicad_pcb format.


## FORMAT

### Header
```
HDR:v1:gen=<generator_name>
BOARD:<width>x<height>
LAYERS:<layer_count>
RULES:clearance=<mm>:track=<mm>:via=<drill>/<annular>
```

### Footprint Placement
```
FP:<ref>:<library:footprint>@(<x>,<y>)[:r<rotation>][:layer]  # value
```
- rotation: 0, 90, 180, 270 (default 0)
- layer: F.Cu or B.Cu (default F.Cu)

### Pad Offsets
```
PADS:<library:footprint>:<pad>@(<dx>,<dy>),<pad>@(<dx>,<dy>),...
```
- One entry per unique footprint library (deduplicated)
- Offsets are relative to footprint origin at rotation=0
- Generated by distill_pcb.py from the .kicad_pcb pad definitions

### Net Definitions
```
NET:<net_name>:<ref>.<pad>,<ref>.<pad>,...
```

### Tracks
```
TRK:(<x1>,<y1>)->(<x2>,<y2>):<width>:<layer>:<net_name>
```

### Vias
```
VIA:(<x>,<y>):<drill>/<annular>:<net_name>
```

### Zones
```
ZONE:<net_name>:<layer>:(<x1>,<y1>)(<x2>,<y2>)...
```

### Comments
```
# Anything after # is a comment
```


## PAD POSITION CALCULATION

Absolute pad positions are computed from FP position + PADS offsets,
adjusted for rotation.

### KiCad rotation convention (CLOCKWISE, verified against KiCad 9.0.7 DRC)

```
rotation 0:   (dx, dy) → ( dx,  dy)   abs_x = fp_x + dx,  abs_y = fp_y + dy
rotation 90:  (dx, dy) → ( dy, -dx)   abs_x = fp_x + dy,  abs_y = fp_y - dx
rotation 180: (dx, dy) → (-dx, -dy)   abs_x = fp_x - dx,  abs_y = fp_y - dy
rotation 270: (dx, dy) → (-dy,  dx)   abs_x = fp_x - dy,  abs_y = fp_y + dx
```

### CRITICAL: rotation is CLOCKWISE, not standard math CCW

KiCad rotations are CLOCKWISE (Y-down screen space). Do NOT use
standard trig (math.cos/sin with positive angle) — that gives CCW
rotation which produces WRONG pad positions for 90° and 270°.

Authoritative implementation: utilities/checks/dpcb_parser.py rotate_pad()
Uses a verified lookup table, not trig.

WARNING: dpcb_viewer.py rotate_pad() uses trig (CCW) — this is internally
consistent for display+grid but gives different positions than KiCad.
When computing pad coordinates for API route commands, use the viewer's
positions (read from the grid/board), do NOT hand-calculate with the
lookup table above. The lookup table is correct for KiCad/.kicad_pcb
coordinates; the viewer uses its own trig-based coordinates internally.

The AI must calculate these explicitly before running verification
checks. Do not estimate or eyeball pad positions.
